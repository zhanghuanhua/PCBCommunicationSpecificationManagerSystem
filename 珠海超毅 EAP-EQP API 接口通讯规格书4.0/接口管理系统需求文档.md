# 珠海超毅 EAP-EQP API 接口管理系统需求文档

## 1. 项目背景

现有《珠海超毅 EAP-EQP API 接口通讯规格书4.0》主要通过 Word 文档维护。接口数量持续增加后，人工编辑容易出现接口编号错误、字段遗漏、JSON 示例格式错误、章节位置错误、格式不一致等问题。

本系统用于将接口规格书从“人工编辑文档”转为“结构化接口数据管理 + 自动生成规格书”。内部人员在系统中维护接口数据，系统统一生成 Word 规格书，并可输出 Markdown 审阅版。最终 Word 可再转换为 PDF，提供给厂商和开发人员查看。

## 2. 建设目标

1. 支持半自动导入现有规格书内容，减少重新录入工作量。
2. 支持维护 `EQP -> EAP` 和 `EAP -> EQP` 两类接口。
3. 支持通过结构化表单新增、编辑、查询接口，避免直接手改 Word。
4. 支持自动生成接口章节内容，包含接口说明、请求参数、返回参数和 JSON 示例。
5. 支持自动导出 Word 规格书和 Markdown 审阅版。
6. 支持后续从本地工具升级为公司内网 Web 系统。

## 3. 使用对象

### 3.1 内部文档维护人员

主要关注：

- 导入现有规格书。
- 新增、修改、作废接口。
- 管理版本和变更记录。
- 导出 Word 规格书。
- 检查字段、示例、编号和格式是否正确。

### 3.2 开发和对接工程师

主要关注：

- 快速查询接口。
- 查看请求字段、返回字段、调用方、提供方。
- 复制 JSON 请求和响应示例。
- 理解接口方向、使用场景和服务描述。
- 查看接口历史变更。

### 3.3 厂商

厂商不直接使用系统，只接收系统生成后的 Word 或 PDF 文档。

## 4. 系统形态

第一阶段建议做成本地网页工具或本地部署工具，满足内部快速落地使用。系统设计保留升级空间，后续可扩展为公司内网 Web 系统，支持多人登录、权限控制、审批流和集中数据管理。

推荐形态：

- 第一版：本地工具，支持单人或小团队维护。
- 后续版：内网 Web 系统，支持多用户协作。

## 5. 核心业务流程

### 5.1 半自动导入流程

1. 用户选择现有 Word 规格书。
2. 系统读取文档章节、接口编号、接口标题、表格和 JSON 示例。
3. 系统生成待确认的接口草稿。
4. 用户在导入确认界面检查并修正接口信息。
5. 用户确认后，接口草稿入库。
6. 系统保留原始导入记录，便于追溯。

说明：由于 Word 表格、中文格式、示例 JSON 可能存在不稳定情况，第一版不要求完全自动解析。系统应优先保证导入后的人工确认体验清晰可靠。

### 5.2 新增接口流程

1. 用户点击“新增接口”。
2. 选择接口方向：
   - `EQP -> EAP`：设备主动上报 EAP。
   - `EAP -> EQP`：EAP 主动下发到设备。
3. 填写接口基础信息。
4. 填写请求参数。
5. 填写返回参数。
6. 系统根据字段自动生成 JSON 示例初稿。
7. 用户修改示例值。
8. 系统执行校验。
9. 用户保存接口。
10. 接口进入当前规格书版本的待导出内容。

### 5.3 导出流程

1. 用户进入导出中心。
2. 选择规格书版本，例如 `4.0` 或后续新版本。
3. 选择导出范围：
   - 全量规格书。
   - 仅新增接口。
   - 指定方向接口。
   - 指定版本变更内容。
4. 系统生成 Word 规格书。
5. 系统生成 Markdown 审阅版。
6. 用户检查 Word 后，可转换为 PDF，提供给厂商和开发人员查看。

## 6. 界面规划

### 6.1 总体布局

系统采用“接口管理工作台 + 新增接口向导 + 导出中心”的界面结构。

主界面分为三个区域：

- 左侧：接口目录和筛选。
- 中间：接口列表。
- 右侧或详情页：接口详情。

这种结构适合同时满足维护人员和开发人员的需求：维护人员可以快速编辑，开发人员可以快速查询。

### 6.2 接口目录区

目录区按接口方向分组：

- `EQP -> EAP`
- `EAP -> EQP`

每个方向下支持以下筛选：

- 接口编号。
- 接口名称。
- Message/API 名称。
- 业务模块。
- 版本号。
- 状态。
- 是否通过校验。

接口状态建议包括：

- 草稿。
- 已确认。
- 已发布。
- 已作废。

### 6.3 接口列表区

接口列表展示字段：

- 接口编号。
- 接口名称。
- 接口方向。
- Message/API 名称。
- 调用方。
- 提供方。
- 版本。
- 状态。
- 校验结果。
- 最后修改时间。
- 最后修改人。

列表支持：

- 搜索。
- 排序。
- 按方向筛选。
- 按版本筛选。
- 按状态筛选。
- 批量加入导出版本。

### 6.4 接口详情页

接口详情页不提供自由编辑整篇 Word 正文，而是提供结构化编辑区域。

详情页包含：

1. 基础信息。
2. 请求参数。
3. 返回参数。
4. JSON 示例。
5. 变更记录。
6. 导出预览。
7. 校验结果。

### 6.5 基础信息区

字段建议包括：

- 接口编号，例如 `EQP-EAP-037` 或 `EAP-EQP-012`。
- 接口名称，例如 `设备状态上报`。
- 接口方向。
- 需求说明。
- 使用场景。
- 接口名称/API 名称，例如 `EQP_EquipmentCurrentStatus`。
- 请求方式，默认 `POST`。
- Content-Type，默认 `application/json`。
- 调用方。
- 提供方。
- 接口服务描述。
- 所属版本。
- 所属业务模块。
- 备注。

系统应根据接口方向自动带出调用方和提供方：

- `EQP -> EAP`：调用方为 `EQP`，提供方为 `EAP`。
- `EAP -> EQP`：调用方为 `EAP`，提供方为 `EQP`。

### 6.6 请求参数区

请求参数分为公共字段和 `Content` 字段。

公共字段建议固定为：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| From | string | 调用接口来源 |
| Message | string | 消息名称，通常为接口 API 名称 |
| DateTime | DateTime | 系统时间，格式为 `yyyy/MM/dd HH:mm:ss` |
| Content | object | 参数内容 |
| RequestId | string | 请求 ID，建议使用毫秒时间格式 |

`Content` 字段由用户维护，支持新增、删除、排序和嵌套。

字段属性建议包括：

- 序号。
- 字段名。
- 类型。
- 是否必填。
- 示例值。
- 描述。
- 父级字段。
- 是否数组。
- 枚举值说明。

### 6.7 返回参数区

返回参数分为公共返回字段和 `Content` 返回字段。

公共返回字段建议固定为：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| Code | string | 结果代码，`0000` 表示成功 |
| Success | bool | 执行是否成功 |
| Msg | string | 提示信息 |
| DateTime | DateTime | 系统时间，格式为 `yyyy/MM/dd HH:mm:ss` |
| Content | object | 返回内容 |
| RequestId | string | 回复请求 ID，与请求的 RequestId 一致 |

`Content` 返回字段可为空，也可维护具体字段。

### 6.8 JSON 示例区

JSON 示例由系统根据字段自动生成初稿，用户主要修改示例值。

示例区包含：

- 请求 URL。
- 请求 JSON。
- 响应 JSON。

系统需要校验：

- JSON 格式是否合法。
- 请求字段是否与参数表一致。
- 返回字段是否与返回值表一致。
- `Message` 是否与接口 API 名称一致。
- `From` 是否与接口方向一致。
- `RequestId` 在请求和响应中是否一致。

### 6.9 导出预览区

导出预览用于展示当前接口生成到规格书中的最终效果。

预览内容包括：

- 接口标题。
- 接口说明表。
- 请求参数列表。
- 返回值列表。
- 日志示例或 JSON 示例。

预览区只用于查看，不建议直接编辑。若发现问题，应返回对应表单或字段表修改。

## 7. 新增接口向导

新增接口建议采用分步向导，降低漏填和误填概率。

### 7.1 步骤一：选择接口方向

用户选择：

- `EQP -> EAP`
- `EAP -> EQP`

系统根据方向自动确定：

- 接口编号前缀。
- 调用方。
- 提供方。
- 默认章节位置。

### 7.2 步骤二：填写基础信息

用户填写接口名称、需求说明、使用场景、API 名称、服务描述和所属版本。

系统自动检查：

- 接口编号是否重复。
- API 名称是否重复。
- 接口方向和调用方/提供方是否匹配。

### 7.3 步骤三：填写请求参数

系统默认带出公共请求字段。用户只维护 `Content` 字段。

### 7.4 步骤四：填写返回参数

系统默认带出公共返回字段。用户只维护返回 `Content` 字段。

### 7.5 步骤五：生成示例

系统根据请求参数和返回参数生成 JSON 示例。

用户可修改：

- 示例值。
- 数组示例条数。
- 空对象或空数组表现。
- 业务枚举示例。

### 7.6 步骤六：校验与预览

系统展示校验结果和导出预览。

校验通过后允许保存。若存在严重错误，应阻止发布和导出。

### 7.7 步骤七：保存入库

保存后接口进入草稿或已确认状态。

## 8. 校验规则

系统需要提供自动校验能力，减少人工错误。

### 8.1 编号校验

- `EQP -> EAP` 接口编号必须以 `EQP-EAP-` 开头。
- `EAP -> EQP` 接口编号必须以 `EAP-EQP-` 开头。
- 编号不得重复。
- 同方向编号建议连续。

### 8.2 方向校验

- `EQP -> EAP` 的调用方必须为 `EQP`，提供方必须为 `EAP`。
- `EAP -> EQP` 的调用方必须为 `EAP`，提供方必须为 `EQP`。
- `From` 字段示例值必须与调用方一致。

### 8.3 字段校验

- 字段名不能为空。
- 类型不能为空。
- 同一层级下字段名不得重复。
- 字段描述不能为空。
- 数组字段需要说明数组项结构。
- 枚举字段需要说明可选值。

### 8.4 JSON 校验

- 请求示例必须是合法 JSON。
- 响应示例必须是合法 JSON。
- 示例字段应与参数表保持一致。
- 请求和响应中的 `RequestId` 应一致。
- `Message` 应与接口 API 名称一致。

### 8.5 导出校验

- 所有待导出接口必须通过基础校验。
- Word 章节顺序应正确。
- 接口标题、编号、表格和示例格式应统一。
- Markdown 审阅版应与 Word 内容一致。

## 9. 导出规格书要求

### 9.1 Word 导出

Word 是第一优先级导出格式，用于内部管理和后续转 PDF。

Word 导出内容应包含：

- 封面信息。
- 文档变更记录。
- 文档概述。
- 约定说明。
- `EQP -> EAP` 接口章节。
- `EAP -> EQP` 接口章节。
- 每个接口的说明、参数、返回值和 JSON 示例。

系统应统一控制：

- 标题层级。
- 表格格式。
- 字体和字号。
- JSON 示例格式。
- 页眉页脚。
- 目录。
- 接口编号顺序。

### 9.2 Markdown 导出

Markdown 用于内部审阅、归档和版本比较。

Markdown 内容应与 Word 主体内容一致，便于快速检查字段和示例。

### 9.3 PDF 交付

第一版可先由 Word 转 PDF。系统界面中保留 PDF 交付检查清单：

- 目录是否更新。
- 页码是否正常。
- 表格是否跨页异常。
- JSON 示例是否完整显示。
- 接口编号是否连续。
- 新增接口是否已出现在正确章节。

后续可扩展为系统直接导出 PDF。

## 10. 数据对象规划

### 10.1 规格书版本

字段建议：

- 版本号。
- 版本名称。
- 创建时间。
- 创建人。
- 状态。
- 变更说明。
- 包含接口列表。

### 10.2 接口

字段建议：

- 接口编号。
- 接口名称。
- 接口方向。
- API 名称。
- 请求 URL。
- 请求方式。
- 调用方。
- 提供方。
- 需求说明。
- 使用场景。
- 服务描述。
- 所属版本。
- 所属模块。
- 状态。
- 备注。

### 10.3 参数字段

字段建议：

- 所属接口。
- 参数类型：请求参数或返回参数。
- 父级字段。
- 序号。
- 字段名。
- 数据类型。
- 是否必填。
- 是否数组。
- 示例值。
- 字段描述。
- 枚举说明。

### 10.4 示例

字段建议：

- 所属接口。
- 请求 URL。
- 请求 JSON。
- 响应 JSON。
- 示例说明。

### 10.5 变更记录

字段建议：

- 所属接口。
- 变更类型。
- 变更前内容。
- 变更后内容。
- 变更说明。
- 修改人。
- 修改时间。

## 11. 权限规划

第一版可先不做复杂权限，但需要预留角色概念。

建议角色：

- 管理员：维护系统配置、版本、导出模板。
- 文档维护人员：新增、修改、确认接口。
- 开发查看人员：查询接口、复制示例、下载文档。
- 审核人员：确认接口是否可发布。

## 12. 第一版范围

第一版建议优先实现：

1. 半自动导入现有规格书。
2. 接口列表和详情查看。
3. `EQP -> EAP` 和 `EAP -> EQP` 接口新增。
4. 请求参数和返回参数结构化维护。
5. JSON 示例生成和校验。
6. Word 导出。
7. Markdown 导出。
8. 基础校验规则。

第一版暂不强制实现：

- 多人在线协作。
- 审批流。
- 系统直接导出 PDF。
- 完整自动解析所有历史规格书。
- 接口自动测试调用。

## 13. 后续扩展

后续可扩展：

- 内网 Web 部署。
- 多用户登录和权限。
- 接口发布审批。
- Word 和 PDF 一键导出。
- 历史版本对比。
- 字段模板复用。
- 接口变更影响分析。
- 与开发测试工具联动。
- 按厂商、设备型号、项目维护多套规格书。

## 14. 成功标准

系统上线后应达到以下效果：

1. 内部不再直接手工编辑新增接口章节。
2. 新增接口可以通过向导完成，并自动生成规格书内容。
3. Word 规格书格式统一，字段和 JSON 示例错误明显减少。
4. 开发人员可以快速查询接口并复制示例。
5. 厂商收到的 Word/PDF 内容清晰、完整、格式稳定。
6. 后续版本升级时，可以通过系统持续维护接口数据。
